One of the most important concepts in distributed systems is understanding consistency models. When data is replicated across multiple nodes, we need to make decisions about how updates propagate and what guarantees we provide to clients.

## The CAP Theorem Refresher

The CAP theorem states that a distributed system can only provide two out of three guarantees:

| Property | Description |
|----------|-------------|
| **Consistency** | All nodes see the same data at the same time |
| **Availability** | Every request receives a response |
| **Partition Tolerance** | System continues operating despite network failures |

In practice, network partitions are unavoidable, so we're really choosing between consistency and availability.

## Strong Consistency

With strong consistency, once a write completes, all subsequent reads will return that value:

```python
# Pseudocode for strongly consistent write
def write(key: str, value: str) -> bool:
    # Wait for all replicas to acknowledge
    acks = []
    for replica in replicas:
        ack = replica.write(key, value)
        acks.append(ack)
    
    # Only return success if ALL replicas confirmed
    return all(acks)
```

The trade-off? Higher latency and reduced availability during partitions.

## Eventual Consistency

Eventually consistent systems prioritize availability:

```python
# Pseudocode for eventually consistent write
def write(key: str, value: str) -> bool:
    # Write to local replica immediately
    local_replica.write(key, value)
    
    # Async replication to other nodes
    background_replicate(key, value, replicas)
    
    # Return immediately
    return True
```

> Eventual consistency doesn't mean "inconsistent." It means that given enough time without new writes, all replicas will converge to the same state.

## Practical Considerations

When designing systems, consider:

1. **Read-your-writes** — Users should see their own updates immediately
2. **Monotonic reads** — Don't show older data after showing newer data
3. **Causal consistency** — Preserve cause-and-effect relationships

---

Choosing the right consistency model depends on your application's requirements. Financial systems might need strong consistency, while social media feeds can tolerate eventual consistency. Understanding these trade-offs is essential for building reliable distributed systems.
